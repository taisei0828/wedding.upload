# セットアップ手順

構成：**フロント（index.html）を GitHub Pages で公開** し、**バックエンド（Code.gs）を Google Apps Script で公開**して、両者をURLでつなぎます。ゲストのGoogleログインは不要です。

```
[ゲストのスマホ] → index.html（GitHub Pages）
                        ↓ 写真をアップロード（fetch）
                   Apps Script Web App（あなたが所有）
                        ↓ 自動保存
                   あなたのGoogleドライブの指定フォルダ
```

---

## 手順1. Googleドライブに保存先フォルダを作る

1. Googleドライブで、写真を集めたいフォルダを新規作成（例：「結婚式ゲスト写真」）。
2. フォルダを開き、URLをコピー。
   `https://drive.google.com/drive/folders/1AbCdEfGhIJKLmnopQRstuVWxyz` の
   末尾の文字列がフォルダIDです。

## 手順2. Google Apps Script を作成・デプロイする

1. [script.google.com](https://script.google.com) にアクセスし、「新しいプロジェクト」を作成。
2. デフォルトで書かれているコードを全部消し、`apps-script/Code.gs` の中身を貼り付けます。
3. `FOLDER_ID` の値を、手順1でコピーしたフォルダIDに書き換えます。
4. 右上の「デプロイ」→「新しいデプロイ」をクリック。
5. 歯車アイコンから種類を選択し「ウェブアプリ」を選ぶ。
6. 設定：
   - 説明：任意（例：結婚式写真アップロード）
   - 次のユーザーとして実行：**自分**
   - アクセスできるユーザー：**全員**
7. 「デプロイ」をクリックし、初回はGoogleアカウントの権限承認（Driveへのアクセス許可）を求められるので許可します。
8. 発行された「ウェブアプリのURL」（`https://script.google.com/macros/s/.../exec` の形式）をコピーします。

> コードを後から修正したときは、「デプロイを管理」→ 既存デプロイの鉛筆アイコン →
> バージョンで「新バージョン」を選んで更新してください（URLは変わりません）。

## 手順3. index.html にURLを設定する

1. `index.html` を開き、以下の行を探します。

   ```js
   const SCRIPT_URL = "https://script.google.com/macros/s/XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX/exec";
   ```

2. `XXXXXXXX...` の部分を、手順2でコピーしたURLに置き換えます。

## 手順4. GitHub Pages で公開する

1. GitHubで新規リポジトリを作成（例：`wedding-photos`）。
2. `index.html` をリポジトリ直下にアップロード（`apps-script/Code.gs` はApps Script側で使うだけなので、リポジトリに含めなくても構いませんが、記録用に一緒に置いてもOKです）。
3. リポジトリの「Settings」→「Pages」を開く。
4. 「Source」を `main` ブランチ / `root` に設定して保存。
5. 数分後、`https://（あなたのアカウント名）.github.io/wedding-photos/` の形式でURLが発行されます。

## 手順5. 動作確認

1. 発行されたURLをスマホで開く。
2. 名前を入力し、写真を1〜2枚選んで送信。
3. Googleドライブの指定フォルダに写真が届いているか確認。
4. 問題なければ、そのURLをQRコードにして招待状や席次表に載せると便利です。

## 容量・大量アップロードの目安（2026年7月時点のApps Script公式クォータに基づく）

| 項目 | 実際の上限 | このサイトでの対応 |
|---|---|---|
| 1リクエストで送れるデータ量 | 50MB（Base64化で約1.37倍に膨らむため、生データは実質36MB程度が上限） | 6MBごとに自動でチャンク分割して送信するため、**ファイルサイズの実質的な上限はほぼありません**（大きい動画ほど分割回数が増えるだけ） |
| 1回の処理時間 | 6分/回 | 1チャンクの処理は一瞬で終わるため問題になりません |
| 同時実行数（自分のアカウント） | 30 | 混雑時に失敗しても自動で待って再送信（最大4回）するため、瞬間的に30人を超えて集中しても数秒〜数十秒待てば通ります |
| **1日の総アップロード件数** | **上限なし**（doPostの呼び出し自体には日次クォータがありません） | 何百件送られても、時間さえかければ確実に処理されます |

つまり「大量に一気に送られる」こと自体は心配いりません。心配すべきはむしろ**Googleドライブの保存容量**です（個人アカウントは通常15GB、Gmail・Googleフォトと共有）。動画は写真の何十倍も容量を使うため、結婚式当日に動画付きで大量に送られると、無料枠だと簡単に埋まる可能性があります。事前に以下のいずれかをおすすめします。

- Google Oneでストレージを増量しておく（100GBで月額250円程度〜）
- 会場のWi-Fi環境が悪い場合に備え、案内文に「動画は短め（1分以内など）でお願いします」と一言添える
- 定期的にドライブの空き容量を確認する

**Q. ゲストがアップロード中にアプリを閉じてしまったら？**
チャンク単位で送信しているので、動画の途中まで送られた分は一時ファイルとしてドライブの `_uploading_tmp` フォルダに残りますが、最後のチャンクが届かないと最終ファイルとして結合されません。`Code.gs` 内の `cleanupOldTempChunks()` を手動実行するか、時間主導型トリガーで1日1回動かしておくと、24時間以上残っている未完了の一時ファイルを自動で削除できます。

**Q. ファイル名は誰のものか分かる？**
はい。保存されるファイル名は「お名前_日時_元のファイル名」の形式になるので、誰がいつ送った写真・動画か一目で分かります。

**Q. セキュリティは大丈夫？**
Webアプリのアクセス権を「全員」にするため、URLを知っていれば誰でも写真・動画を送信できます（逆に、送るためだけならこれが必須です）。ただし送信専用（doPost）のみを実装しており、フォルダ内の既存ファイルの閲覧・削除などはできない設計です。URLを不特定多数に公開しない（招待客のみに共有する）運用にすれば十分安全です。

## 集めた写真・動画を後でみんなに共有するには（Googleフォトの共有アルバム）

はい、**Googleフォトの共有アルバムで問題ありません**。運用イメージは次の通りです。

1. 結婚式後、Googleドライブの保存フォルダに集まった写真・動画をGoogleフォトの新規アルバムに追加する（ドライブの写真をフォトのアルバムへ「追加」できます）。
2. アルバムの共有設定で「リンクの共有」をオンにする。
3. 「共同編集」はオフのままにしておく（オンにすると、リンクを知っている人が新たに写真を追加できてしまうため、あなたが集めた写真だけを見せたいなら閲覧専用のままで十分です）。
4. 発行されたリンクをゲストに送る。

**セキュリティについての正確な理解**
「リンクを知っている人だけがアクセスできる」というのはその通りですが、これは論理的な非公開設定というより「推測されにくいURLを知っている人だけが見られる」という仕組み（Googleの公式ヘルプでも「推測できないURL」と説明されています）です。つまり：

- パスワード保護のような強固な仕組みではなく、リンクが転送・スクリーンショットなどで広まれば、意図しない人にも見られる可能性があります。
- 個人情報が写り込んだ写真など、見られたくない相手がいる場合は、招待客だけに限定してリンクを送る運用にしてください。
- 共有を終了したいときは、アルバムの共有オプションから「リンクの共有」をオフにすればいつでも無効化できます（リンクも新しく発行し直せます）。
- 結婚式の集合写真程度であれば、一般的にはこの方式で十分実用的とされています。

より厳格に「招待した人しか見られない」ようにしたい場合は、リンク共有ではなく、ゲスト一人ひとりのGoogleアカウント（メールアドレス）を個別にアルバムへ招待する方法もありますが、手間が増えるぶん、通常はリンク共有で運用されるケースがほとんどです。
